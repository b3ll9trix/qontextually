We provide a simulated enterprise dataset including email, CRM, HR, policy documents, collaboration/workspace data, IT service data, and business records.     - a virtual file system that documents the business: static data (employees, customers, products), procedural knowledge (processes, SOPs, rules), and trajectory information (tasks, projects, progress)
    - explicit references both inside and outside the graph: links to other files, and links to the underlying source records where the information comes from
    - interface(s) that enable AI systems to efficiently retrieve context and both business users and AI systems to inspect, validate, edit, and extend the company memory

## 🔁 Pillar 1.5 — Entity Resolution & Rolling Updates (Hours 8–11) — the demo-winner

### Three-tier entity resolution

- [ ]  **Tier 1** — exact match (lowercase + strip + same type)
- [ ]  **Tier 2** — embedding similarity (text-embedding-3-small, 1536-dim, or voyage-3-lite)
- [ ]  Embeddings stored in `entity_embeddings` virtual table on every new entity
- [ ]  Entity text built from name + type + properties + source span
- [ ]  Cosine sim ≥ 0.90 → auto-merge; 0.75–0.90 → queue for human; < 0.75 → new entity
- [ ]  **Tier 3** — ambiguous matches inserted as `conflicts` rows with `conflict_type='entity_match'`
- [ ]  Within-document deduplication before resolution

### File-level rolling updates

- [ ]  `ingest.py` polling loop (every 30s, scan data dir)
- [ ]  Diff current files against `tracked_files` table (path + sha256)
- [ ]  **ADD handler** — extract + insert with source pointer
- [ ]  **DELETE handler** — remove `triple_sources` rows for that source
- [ ]  Orphan logic — triple with zero remaining sources → delete or mark `status='orphaned'`
- [ ]  **MODIFY handler** — delete-then-add composition
- [ ]  Orphaned entities (zero triples) cleaned up too
- [ ]  Every mutation logged to `audit_log`

### Live ingest demo path

- [ ]  Drop a new file → graph grows within 30s
- [ ]  New file conflicts with existing fact → conflict surfaces
- [ ]  Delete one of multiple sources → triple stays alive (multi-source provenance demo)
- [ ]  Delete last source → triple orphaned/removed

## ⚖️ Pillar 2 — Conflict Resolution (Hours 11–15)

### Detection · h11–12

- [ ]  On insert: check same `(subject_id, predicate)` with different object where predicate is functional
- [ ]  Mark conflicting triples `status='conflicted'`
- [ ]  Conflict row created with both triple ids and `conflict_type='fact'`
- [ ]  Logged to `audit_log`

### Heuristic resolution · h12–13

- [ ]  Authority pulled from `source_types` table (HR=1.0, CRM=0.8, policy=0.7, email=0.4, chat=0.3, unknown=0.5)
- [ ]  Recency score (normalized 0–1)
- [ ]  Per-source score = `authority × confidence × recency` (multiplicative)
- [ ]  Per-triple score = sum across all supporting `triple_sources`
- [ ]  Auto-resolve if `winner_score − loser_score > 0.3`
- [ ]  Mark winner `status='active'`, loser `status='superseded'`, log `scores_json`
- [ ]  Otherwise queue for human; reason logged either way

### Human-in-loop UI · h13–15

- [ ]  FastAPI app running (or Streamlit if faster for you)
- [ ]  Pending conflicts list page
- [ ]  Detail view — both triples + sources + highlighted spans
- [ ]  Score breakdown shown (authority × confidence × recency per source)
- [ ]  Pick A / Pick B buttons working
- [ ]  Free-text "actually it's X" field (cuttable)
- [ ]  Resolution + reason written to DB on click, logged to `audit_log`
- [ ]  **Bonus:** "discovered predicates" panel — promote auto-added predicates to functional with one click
