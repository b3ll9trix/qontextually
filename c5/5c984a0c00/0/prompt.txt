POST /vocabulary/{predicate}/dismiss

## Screens to build
### Navigation / Layout
- **Sidebar** (left, 240px): Dashboard, Conflicts (with badge count), Vocabulary (with badge count), Entities, Sources, Graph. At the bottom: backend health dot (green/red) from `/health`.
- **Top bar**: "Qontextually" wordmark left, app version/env right.
- **Main area**: route content.
- Dark mode only. Use shadcn's default dark theme (slate background, sparing use of blue and emerald for accent).
- Use monospace font for IDs (`e_ec0c4bb0e690`, `triple #4722`), sans-serif for everything else.
### Screen 1 — Dashboard (`/`)
Top counters in a responsive grid: Entities, Triples, Sources, Predicates. Each counter is a card with the big number, the breakdown below (e.g. Predicates: "14 seeded, 2613 auto, 4 merged"), and `last_extraction_at` in a small timestamp at the bottom.
Below the counters, **two prominent action cards** side by side:
1. **Pending Conflicts** (red accent if count > 0): "N conflicts need your review → Review". Button takes you to `/conflicts`.
2. **Discovered Vocabulary** (amber accent): "N predicates awaiting promotion → Review". Takes you to `/vocabulary`.
Below that, a small "Entities by type" horizontal bar chart (use recharts). Then an "avg sources per triple" metric with a short caption: "The graph is densely provenanced — most facts have more than one supporting source."
All data from `GET /stats`. Poll every 30s.
### Screen 2 — Conflict Queue (`/conflicts`)
**List view** at top: table of pending conflicts with columns: Subject entity | Predicate | Candidate A (value preview) | Candidate B (value preview) | Auto-suggestion | Created. Click a row to open the detail pane on the right side.
**Detail pane** (right, fixed 50%+ width on desktop, full-screen on mobile): the meat of the HITL moment.
Show the subject entity card at top (name + type + `id` in monospace).
Then two side-by-side candidate panels labeled **A** and **B**. Each shows:
- The value (literal string or object entity) in a large card
- A score breakdown widget: three small bars for authority × confidence × recency, multiplied, total score
- The list of supporting sources: for each source, show `source_type` badge (color-coded: hr=blue, crm=indigo, policy=violet, ticket=orange, email=slate, chat=gray), `document_path` truncated, `extracted_at`, and an expandable "view snippet" that reveals the `snippet` text.
Below the two panels, an **auto-suggestion banner**: "Our scoring suggests **Candidate A** wins (reason: ${auto_resolution_hint.reason}). You can accept this, override, or reject both."
Action bar at the bottom of the detail: three primary buttons:
- **Accept A** (green)
- **Accept B** (orange)
- **Neither** (gray)
Plus an optional single-line text input "Note (optional)". Clicking any button sends `POST /conflicts/{id}/resolve` with the chosen winner, then moves to the next conflict in the queue.
Keyboard shortcuts: `A` → accept A, `B` → accept B, `N` → neither. Show these as tiny badges on each button.
Empty state: a cheerful checkmark illustration with text "No pending conflicts. The system is caught up."
### Screen 3 — Discovered Vocabulary (`/vocabulary`)
Two-pane layout. **Left**: list of predicates awaiting review, sortable by occurrence count (default descending). Each row: predicate name (monospace), usage count, nearest canonical + cosine score, a small color dot for similarity (green ≥0.9, amber 0.85–0.9, gray <0.85).
**Right**: detail for the selected predicate.
- Name (big, monospace)
- Usage count + description (if any)
- **Nearest canonical match**: card showing `{ predicate, cosine, occurrence_count }`. Subtitle: "Similarity is above review threshold but below auto-merge. Your call."
- **Sample triples**: render 3–5 real triples in a readable format: `subject_name — predicate → object` (with object being the value from sample_triples[].object)
- Action bar at bottom:
  - **Merge into canonical** (primary) — with a subtle dropdown to override the target predicate (defaults to `nearest_canonical.predicate`). On click: `POST /vocabulary/{predicate}/merge` with `into` set.
  - **Promote as canonical** (secondary) — opens a small modal: "Mark as functional?" checkbox + description field. On submit: `POST /vocabulary/{predicate}/promote`.
  - **Dismiss** (tertiary, ghost style) — calls `POST /vocabulary/{predicate}/dismiss`.
Design callout above the list: a small banner explaining why 0.85+ pairs are here — exact text to show: *"These predicates scored between 0.75 and 0.95 cosine similarity to an existing canonical predicate. Our system treats this band as ambiguous: lexical variants (works_at ≈ works_for) usually cluster ≥ 0.95, but polarity-flipped pairs (seeks_advice vs offers_advice) can also score here. Your call is the final signal."*
### Screen 4 — Entity Browser (`/entities`)
Split layout: **left column (320px)** — search + filter + results list; **right** — entity detail.
Left column:
- Search input (debounced 300ms, updates `q` param on `/entities`)

- Pagination or infinite scroll with `react-query`'s `useInfiniteQuery`.
Right column (when an entity is selected, call `GET /entities/{id}`):
- Header: name (big), type badge, id (monospace small)
- Properties table: key / value pairs in a clean grid
- Aliases: chips with `alias_type` as label, asterisk for `is_primary`
- Tabs for "Outgoing (N)" and "Incoming (N)" triples. Each list shows predicate + object, clicking the triple opens a provenance drawer that calls `GET /triples/{id}/provenance`.
- A button in the header: "Open in Graph View" — navigates to `/graph?center={id}`.
### Screen 5 — Source Browser (`/sources`)
Table layout with filters across the top: `source_type` dropdown (all / hr / crm / policy / ticket / email / chat / unknown), date range picker (on `extracted_at`).
Table columns: Source type badge | Document path (truncated, full path on hover) | Extracted at | Triple count | Entity count | Actions.
Click a row to open a modal with the full source view from `GET /sources/{id}`: complete `raw_text` in a scrollable code-style pane on the left (monospace, whitespace-pre), and on the right, two lists: "Contributed entities" and "Contributed triples" — each item a clickable link back to the Entity Browser.
Pagination at the bottom.
### Screen 6 — Graph Visualization (`/graph`)
A 3D force-directed graph using `react-force-graph-3d`.
**Critical constraints** (tell Lovable exactly this):
- The backend returns at most **500 nodes**. Do not attempt to render more. If user requests more detail, they should re-center on a specific entity.
- The default view (no `center` param) shows the 300 most-connected entities as an overview.
- Users can type an entity name into a search box; picking a result sets `center` and re-fetches.
- Node color by `type` (consistent with the entity-type badge colors used elsewhere).
- Node size proportional to `degree` (clamp: min 4, max 20).
- Edge thickness proportional to `source_count` (clamp: min 1, max 5) — this visually shows multi-source provenance.
- On node hover: tooltip with name, type, degree.
- On node click: open a panel on the right with the entity card (minimal version: name, type, properties) and a button "Inspect in Entity Browser" that navigates to `/entities?q={name}`.
- On edge hover: tooltip with predicate and source_count.
Controls overlay (top-right corner, floating):
- Search input: "Center on entity..."
- Depth selector: 1, 2, 3 (default 2)
- Max nodes selector: 100, 300, 500 (default 300)
- "Reset camera" button
Performance note for Lovable: use `react-force-graph-3d`'s `cooldownTicks={100}` to stop simulation once stable. Don't re-mount the graph on every filter change — use its `graphData` prop.
Empty state if no data: a clean panel explaining: "The graph is empty or the backend is unreachable. Check health indicator."

Styling and UX rules
- Dark mode only, using shadcn/ui's default dark theme.
- Density: moderate — prefer whitespace over cramming, but don't waste vertical space.
- Monospace (font-mono): all IDs (e_ec0c4bb0e690, triple #4722, source #4332), all predicate names, all document paths. Everything else sans-serif.
- Color palette for source types: consistent across all screens. hr=blue-500, crm=indigo-500, policy=violet-500, ticket=orange-500, email=slate-500, chat=gray-500, unknown=neutral-500.
